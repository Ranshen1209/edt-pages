Why reverse proxy mode? What is PROXYIP?


                    

                        In simple terms:



                        The problem:

                        edgetunnel is deployed on Cloudflare Workers, but Workers cannot directly access Cloudflare-owned services.
It is like Cloudflare trying to see itself through its own eyes.



                        Solution:

                        To solve this, use an external server (ProxyIP, SOCKS5, or HTTP) as a relay to access Cloudflare services.
Think of ProxyIP/SOCKS5/HTTP as a mirror that lets Cloudflare reach itself indirectly.



                        Practical effect:
                        
    
                            
  • When visiting non-Cloudflare CDN sites such as YouTube and Google, your IP is determined by the preferred IP.
    • 
                            
  • When visiting sites hosted by Cloudflare CDN such as Twitter and ChatGPT, your IP is determined by reverse proxy mode (PROXYIP/SOCKS5/HTTP).
    • 
                        

                    

                

            


            
            

                

                    
Why reverse proxy mode? What is PROXYIP?

                    
Why is reverse proxy mode needed?

                    

                        According to Cloudflare Workers TCP Sockets documentation , the following technical limitation exists:
                    

                    

                        ⚠️ Outbound TCP sockets to Cloudflare IP ranges
                            ↗ are temporarily blocked, but will be re-enabled shortly.
                    


                    

                        This means Cloudflare Workers cannot connect directly to Cloudflare-owned IP ranges. A third-party cloud server is needed as a relay.
                    


                    

                        Our edgetunnel is a proxy service deployed on Cloudflare Workers. According to Workers restrictions, Worker nodes cannot directly connect to Cloudflare-owned IP ranges. This means:
                    

                    
    
                        
  • Can access non-Cloudflare CDN sites normally
    • 
                        
  • Cannot directly access sites hosted by Cloudflare CDN
    • 
                    

                    

                        Reverse proxy mode solves this by using a third-party server as an intermediary.
                    


                    

                        

                            

                                
Cloudflare Workers

                                
Send request

                            

                            
→

                            

                                
PROXYIP/SOCKS5/HTTP server

                                
Third-party proxy

                            

                            
→

                            

                                
Cloudflare service

                                
Target service

                            

                        

                        

                            Reverse proxy Cloudflare port 443 through a third-party server so Workers can access Cloudflare services.
                        

                    


                    
Practical effect

                    

                        Your apparent IP location depends on the target site type:
                    

                    
    
                        
  • When visiting non-Cloudflare CDN sites such as YouTube and Google, your IP location is determined by the preferred IP.
    • 
                        
  • When visiting sites hosted by Cloudflare CDN such as Twitter and ChatGPT, your IP location is determined by ProxyIP/SOCKS5/HTTP.
    • 
                    

                

            


            
            

                
                
            


            
        

    


    
    

        

            

            
            

                
                
                
            


            
            

                

                    

                        
What is ECH?


                        

                            ECH (Encrypted Client Hello) is a TLS feature used to
                            encrypt domain information that would otherwise be exposed in plaintext. 
                        


                        

                            For a deeper technical explanation, see Cloudflare’s official blog:

                            👉 https://blog.cloudflare.com/zh-cn/announcing-encrypted-client-hello/
                        


                        
Background


                        

                            When using WS + TLS nodes as proxies:
                        


                        
    
                            
  • The actual communication content is already encrypted by TLS, and GFW
                                cannot see the specific content you visit
                            
    • 
                            
  • But during the TLS handshake, one key piece of information may still be exposed:SNI
                            
    • 
                        


                        

                            SNI is the node camouflage domain (HOST)
                        


                        

                            In other words:
                        


                        

                            GFW may not know what you are visiting, but it can know that you are proxying and which domain you connect to.
                        


                        
What problems can this cause?


                        

                            This is why these scenarios often fail:
                        


                        
    
                            
  • v2rayN batch real-link latency tests
    • 
                            
  • Clash policy group automatic node selection
    • 
                        


                        

                            Typical symptom:
                        


                        

                            all nodes instantly show -1 and become unavailable
                        


                        

                            The reason is usually not that nodes are actually down, but that Carriers or GFW interfere with proxy connections by blocking access to the node domain.. 

                            Because this blocking is usually automated and prone to false positives, nodes may recover after some time, then repeat the cycle.
                        


                        
What ECH does


                        

                            The core purpose of ECH is:
                        


                        

                            to encrypt SNI, which is the node domain.
                        


                        

                            After enabling ECH:
                        


                        
    
                            
  • ✅ GFW
                                the real node domain cannot be read.
                            
    • 
                            
  • Externally observed domains are shown as:cloudflare-ech.com
                            
    • 
                        


                        

                            This fundamentally blocks precise node filtering by domain.. 
                        


                        
Can ECH be blocked?


                        

                            In theory, GFW only needs to block directly cloudflare-ech.com to do it.
                        


                        

                            But doing so would:
                        


                        
    
                            
  • affect many legitimate websites and services that use ECH
    • 
                            
  • increase blocking cost and side effects
    • 
                        


                        

                            Therefore,how long ECH remains effective depends on GFW policy tradeoffs, and it is still effective for now.
                        

                    

                


                

                    

                        
How to use ECH


                        

                            Using ECH is simple: just ECH Settings for enabled then update the subscription.
                        


                        
Clients that support ECH


                        
Windows: 

                        
    
                            
  • 
                                v2rayN ≥ v7.17.0
    
                                https://github.com/2dust/v2rayN/releases
                            
    • 
                        


                        
Android: 

                        
    
                            
  • 
                                v2rayNG ≥ v2.0.0
    
                                https://github.com/2dust/v2rayNG/releases
                            
    • 
                        


                        
Clash.Meta (mihomo core):

                        
    
                            
  • 
                                All clients using the clash.meta / mihomo core support ECH.
                            
    • 
                            
  • 
                                ⚠️ DNS support is required.
                            
    • 
                        


                        
Notes for Clash users


                        

                            The current subscription config already includes the DNS settings required by ECH.. 
                        


                        

                            If this happens:
                        


                        
    
                            
  • ECH is enabled
    • 
                            
  • the subscription has been updated
    • 
                            
  • ECH nodes still cannot be used
    • 
                        


                        

                            Check:
                        


                        

                            whether the original DNS settings were overwritten when updating the subscription.
                        


                        

                            Recommendation:do not overwrite the DNS settings included in the subscription.
                        

                    

                


                

                    

                        
🔍 How to confirm ECH is active?


                        

                            You can use a simple and direct method to verify whether ECH works.
                        


                        
Verification steps


                        
    
                            
  1. Open the node 「⚙️
                                    Detailed Configuration」
    2. 
                            
  2. In HOST
                                manually enter a blocked domain from your current project, for example: *.workers.dev
                            
    3. 
                            
  3. After saving the configuration, update the subscription again.
                            
    4. 
                            
  4. In the node list, find:
                                
      
                                    
    • - nodes whose HOST is *.workers.dev
                                        
      • 
                                    
    • - then try connecting to that node.
      • 
                                
    
                            
    5. 
                        


                        
How to interpret the result


                        
    
                            
  • 
                                If the node connects normally,
    
                                it means ECH is working successfully, the real blocked domain is hidden and externally shown as cloudflare-ech.com
                            
    • 
                            
  • 
                                If the node cannot connect,
    
                                ECH is not active, or the client version / DNS configuration has a problem.
                            
    •

Disable TLS transport encryption

                

                    You are about to disable Shadowsocks TLS transport encryption. Confirm the deployment environment first to avoid losing node connectivity.
                

                

                    Before disabling TLS, confirm these three points:
                    
    
                        
  1. The project must be deployed on Workers(not Pages).
    2. 
                        
  2. 
                            In Cloudflare, disable "Always Use HTTPS”
                            
    
                                
                                SSL/TLS > Edge Certificates > Always Use HTTPS > disable
                            
    
                            or HOST Use the *.workers.dev domain assigned to the project directly.
                        
    3. 
                        
  3. 
                            The port for the preferred IP/domain must be an HTTP type port, for example:
                            80、8080、8880、2052、2082、2086、2095. 
                        
    4. 
                    

                

                

                    
Transport and encryption notes

                    
After TLS is disabled, traffic is transported as plaintext HTTP and no longer has TLS wrapping/camouflage for the link.

                    
The node is still encrypted by Shadowsocks AEAD (aes-128-gcm / aes-256-gcm), so traffic payloads remain encrypted.

                    
Summary: final traffic will transport in plaintext form encrypted data.
